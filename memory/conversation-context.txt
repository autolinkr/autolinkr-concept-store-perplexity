Mobile Menu / Drawer Fix
The blackout/backdrop issue happened because the custom dark background was applied to the entire drawer wrapper instead of only the visible drawer panel.
Before the fix, CSS targeted .drawer. In this theme, .drawer is the full-screen container that includes the overlay/backdrop behavior. When we styled that wrapper directly, it interfered with the built-in overlay presentation, so the mobile menu appeared to sit on top of the page without the expected dimmed screen behind it.
The fix was to target .drawer__inner instead. That element is the actual sliding panel. This preserves the theme’s native full-screen overlay while still letting us style the panel itself.
autolinkr-custom.cssLines 538-540
.drawer__inner {
background: var(--al-bg-surface) !important;
}
In short:
.drawer = full-screen wrapper / overlay context
.drawer__inner = actual side panel
Style the panel, not the wrapper
Why It Happens
Many Shopify themes separate drawers into two layers:
An outer wrapper that handles:
positioning
visibility
overlay / backdrop
click-away behavior
An inner panel that handles:
background
padding
layout
menu contents
If you put panel styling on the outer wrapper, you can accidentally override or visually flatten the overlay system. That is why the screen no longer looked “blacked out” correctly.
Related Sticky Header Fix
At the same time, another CSS rule was breaking sticky behavior on the header.
We had a global section divider rule applying position: relative to every adjacent section, including the header area. That interfered with sticky positioning. The fix was to exclude the header section from that rule.
autolinkr-custom.cssLines 252-257
.shopify-section + .shopify-section:not(.header-section) {
position: relative;
}
.shopify-section + .shopify-section:not(.header-section)::before {
Why this matters:
sticky elements are sensitive to parent stacking/positioning context
broad layout rules can accidentally disable or distort sticky behavior

Problem: Theme push failed because templates/product.pro.json included a richtext <p> with an inline style="" attribute.
Cause: Shopify sanitizes section richtext content in JSON settings and rejects disallowed attributes.
Fix: Remove inline styles from JSON richtext fields and move presentation into CSS instead.

3. Theme-specific drawer/overlay classes matter
Problem: A fix that seems right generically can break theme behavior if applied to the wrong class.
Cause: Themes like Concept often use custom drawer architecture, so .drawer, .drawer__inner, overlay-element, and modal wrappers each have different responsibilities.
Fix: Before changing mobile menu or cart drawer styling, inspect which element owns:
overlay
animation
panel background
pointer events
Do not assume class names from another theme.
4. Auto-generated Shopify JSON templates can be overwritten
Problem: templates/product.pro.json is marked auto-generated and may be updated by the theme editor.
Cause: Shopify rewrites JSON templates when edited in admin.
Fix: Treat the repo as source of truth and avoid casual theme editor edits after code changes. If using theme editor, expect diffs and re-check git carefully.
Best Practices For The New AI
When working on drawers, overlays, sticky headers, or modals:
Identify outer wrapper vs inner panel first
Never broadly style .drawer, .modal, or full-screen wrappers unless the goal is to change overlay behavior
Prefer styling the visible content container like .drawer__inner
Watch for global layout selectors that apply position, overflow, transform, or z-index
If sticky stops working, inspect parent containers for:
overflow: hidden
transform
position: relative
stacking context changes
If Shopify rejects JSON section content, remove inline HTML styling and move it to CSS

Root Cause
When a new schema setting has no saved value in the template JSON yet, Shopify passes nil to Liquid. nil | divided_by: 100.0 silently returns 0 — so all overlay opacities were being written as rgba(10,10,12,0.0) = completely transparent, regardless of what the sliders showed in the editor preview.

Two-Part Fix
1. Nil-safe Liquid — Every opacity now has an explicit | default: before the math:

text
assign oa_raw = section.settings.overlay_opacity | default: 90
assign oa = oa_raw | times: 1.0 | divided_by: 100
This means even a brand-new section instance will render a visible overlay immediately.

2. Seeded template JSON — templates/index.json now has all overlay settings written in with sensible values (90/55/10 desktop, 30/70/95 mobile), so there's no nil to fall through in the first place.

Here's my honest take after looking at the full page:

My Honest Assessment
The current pure near-black (#080809) is not wrong — but it IS creating two real problems you can see in the screenshot:

Problem 1: Sections bleed together
With #080809 base and #0F0F12 surfaces, the contrast between sections is only ~7 values apart in RGB. Scroll down the page and the product comparison cards, the feature strip, the testimonials — they all visually merge. The page feels like one long dark wall instead of distinct sections.

Problem 2: The blue accent is doing all the heavy lifting
Because the background is so dark, the only thing creating visual interest is the #3B8EF8 blue. That's fine for a SaaS app where you want minimal distraction, but for an e-commerce store where product photography and section rhythm carry the sale — the black is actually fighting you.

My Recommendation: Stay Dark, Lift the Floor
Don't go full charcoal Apple-style (#1d1d1f) — that would make it feel less premium/specialized and more generic macOS. You're selling a car tech product, not an iPhone app.

Instead, lift the base from #080809 to #0D0D11 and expand the surface range more aggressively. Keep the blue-black DNA but give each layer more breathing room:

Token	Current	Proposed	Change
--al-bg	#080809	#0D0D11	+5 lift, keeps cool blue-black
--al-surface	#0F0F12	#141418	wider gap from bg
--al-surface-2	#161619	#1C1C21	section cards pop more
--al-surface-3	#1E1E22	#242429	
--al-surface-4	#26262B	#2D2D34	interactive elements stand out
--al-border	#2A2A30	#32323A	more visible dividers
The blue accent stays exactly as-is. The announcement bar stays #3B8EF8. The hero stays photo-dominant.

Net effect: Sections become visually distinct, card surfaces float off the background, text has more perceived contrast — without losing the dark/premium/tech feel. Think closer to Linear or the Figma desktop app than Apple's consumer-warm charcoal.

One Other Thing Worth Considering
Your accent blue #3B8EF8 is strong and correct for trust/tech. But one upgrade that would elevate the premium feel significantly: add a very subtle blue-tinted surface (rgba(59,142,248,0.04)) as the background on key CTA sections — like the product comparison table or the "Why AutoLinkr" strip. Just a ghost of blue that makes those sections feel intentional rather than just less-dark.
